# 开发原则浓缩版

## 🔴 核心原则

**框架思维，不是想到哪改到哪。**

任何改动先确认：
- 属于哪个 Phase
- 解决什么问题
- 验收标准是什么

---

## 强制流程（8 步）

1. **RULES.md** — 读规则
2. **status.md** — 读当前状态
3. **PRD** — 写需求文档（新功能必须）
4. **TASK** — 写任务分解（改动 > 1 文件必须）
5. **实现** — 写代码 + 写测试
6. **验证** — 跑测试 + 人工验证
7. **文档同步** — 更新所有受影响文档
8. **提交** — commit + push + check

**小改可跳 3-4，但 6-8 永远不跳。**

---

## MEMORY.md 里的红线规则

| 规则 | 说明 |
| ---------------------- | --------------------------------------------------- |
| 🔴 **写代码必须用 Opus** | 包括子代理 |
| 🔴 **文档链必须走完** | RULES → PRD → ARCH → TASK → 实现 → VERIFY → 文档同步 → 提交 |
| 🔴 **代码写完先 VERIFY 再 push** | verify.sh(V1-V6) + 文档同步 + sync.sh |
| 🔴 **一次性闭环提交** | 代码+文档+日志同一轮全部 commit，不留"待提交" |
| 🔴 **工程化产品标准** | 测试/lint/CI/代码规范/模块边界/错误处理 |
| 🔴 **研究必须闭环到代码** | RULES.md → CI 强制 → 定期检查 |

---

## 其他约束

- **复用优先**，不造轮子
- **PRD 是约束**不是摆设
- 教训写 **lessons.md** + 映射到自动检查
- 重大决策写 **decisions.md**
- **密钥不写明文**，统一存 `.env`

---

## 一句话总结

**先想清楚再动手，做完验证再提交，全程留痕可追溯。**
