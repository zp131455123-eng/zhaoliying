## 类别1：安全/密钥

### 教训1: 密钥不能写明文

**原则**：
- 配置文件（TOOLS.md, config.json）只记录环境变量名
- 真实密钥只存在 `.env` 或系统环境变量中
- `.env` 必须在 `.gitignore` 第一行

**示例**：
```markdown
# ❌ 错误
API_KEY = "sk-1234567890abcdef"

# ✅ 正确
API_KEY = ${OPENAI_API_KEY}  # 环境变量
```

**教训**：
- **密钥泄露不可逆，预防为主**
- **commit 前必须检查敏感信息**

## 类别2：时区/配置

### 教训2: 时区必须明确标注

**现象**：
- Cron 任务执行时间不对
- 日志时间戳混乱

**原因**：
- Cron 默认使用 UTC 时区
- 不同系统/服务的默认时区不同

**解决方案**：
```bash
# Cron 配置（UTC 时区）
# 北京时间 08:00 = UTC 00:00
0 0 * * * /path/to/script.sh

# 脚本内部指定时区
export TZ="Asia/Shanghai"
```

**教训**：
- **所有时间配置必须标注时区（UTC / Asia/Shanghai）**
- **日志时间戳统一格式：YYYY-MM-DD HH:MM +TZ**

## 类别3：记忆管理

### 教训3: MEMORY.md 要保持精简

**原则**：
- MEMORY.md ≤ 150 行
- 只保留活跃项目和热记忆
- 超过 7 天且已完结的内容归档

**归档流程**：
1. 将完结项目移至 `memory/archive-index.md`
2. 将旧的"最近学习/发现"移至对应日志文件
3. 更新 MEMORY.md 的"最后更新"时间戳

**教训**：
- **热记忆要热，定期清理才能快速加载**
- **归档不是删除，是换个地方存（语义搜索仍能找到）**

## 类别4：文档同步

### 教训4: 改了架构必须同步文档

**场景**：
- 修改了 API 接口
- 调整了目录结构
- 更新了配置格式

**必须同步**：
- 项目的 README.md
- 相关的 API 文档
- MEMORY.md 的索引表格
- memory/projects.md 的项目状态

**教训**：
- **代码和文档要同步 commit**
- **commit 前自检：新会话的我能看懂吗？**

## 类别3（补充）：记忆管理

### 教训5: MEMORY.md 必须是轻量索引，不能堆内容

**现象**：MEMORY.md 越写越长，新会话加载慢，信息找不到  
**原因**：把内容直接写进索引文件，而不是指向子文件  
**正确做法**：
- MEMORY.md ≤ 80行，只放索引表格 + 红线规则 + 当前活跃
- 项目详情 → `memory/projects.md`
- 踩坑 → `memory/lessons.md`
- 每日事件 → `memory/logs/YYYY-MM-DD.md`
- 向量搜索覆盖所有 .md，细节随时能找到

## 类别5：数据操作

### 教训6: 原始数据缺失时不能猜测填写

**事件**：杨浩的平台/账号原始 Excel 没有记录，我自己猜着填了备注  
**结果**：被老板批评  
**原则**：
- 原始数据没有 = 空着，标记"原始数据未记录"
- 不能用"可能是"、"推测是"来填充数据
- 发现缺失 → 如实报告给老板，等待补充

### 教训7: 每次修改数据后必须跑验证脚本

**做法**：`python validate.py` 只报错误和警告，不自动补全  
**原则**：验证脚本是最后一道防线，不能跳过

### 教训8: 网络配置修改必须老板确认
